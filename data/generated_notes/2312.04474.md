# Summary

This paper introduces [[Chain of Code (CoC)]], a novel approach to enhance [[language model]] reasoning by integrating both executable code and linguistic simulation. It addresses the challenge of applying code-driven reasoning to semantic tasks that are difficult to fully express in code. CoC achieves this by allowing LMs to generate [[pseudocode]] for [[semantic sub-tasks]], which are then "emulated" by the LM itself (termed [[LMulator]]) when a standard interpreter cannot execute them. The main finding is that CoC significantly outperforms [[Chain of Thought]] and other baselines, achieving state-of-the-art performance on [[BIG-Bench Hard]] tasks.

# Main Contribution

- Proposing [[Chain of Code (CoC)]] to improve [[language model]] [[code-driven reasoning]] by combining code execution and [[LMulator]] simulation.
- Introducing the '[[LMulator]]' concept, enabling [[language models]] to simulate outputs of undefined [[pseudocode]] for [[semantic sub-tasks]].
- Demonstrating that [[CoC]] outperforms [[Chain of Thought]] and other baselines across various benchmarks, including a 12% gain on [[BIG-Bench Hard]].
- Achieving state-of-the-art performance on [[BIG-Bench Hard]] tasks, surpassing average human raters.

# Research Problem

[[Language models]] (LMs) have shown impressive reasoning abilities, particularly with [[Chain of Thought]] prompting, but often struggle with tasks requiring precise numeric or symbolic reasoning. While prompting LMs to write and execute code can address these limitations by leveraging the [[syntactic structure]] and [[algorithmic computations]] of code, this approach faces a significant hurdle: many [[semantic tasks]] are exceedingly difficult, if not impossible, to fully express in executable code. This paper aims to bridge this gap, seeking a method that combines the precision of code execution with the flexibility of linguistic reasoning, without requiring every intermediate reasoning step to be strictly executable by a traditional interpreter.

# Method

[[Chain of Code (CoC)]] is presented as a two-step process: Generation and Execution. In the Generation step, given a problem, the [[language model]] generates reasoning substeps structured as code. This code can range from explicit, runnable code to flexible [[pseudocode]] or even natural language, providing a framework for problem-solving.

The core innovation lies in the Execution step, which employs an [[LMulator]]—an LM-augmented code emulator. When the generated code is encountered, a standard [[code interpreter]] (e.g., Python) attempts to execute each line. If the interpreter encounters a line of [[pseudocode]] or an undefined [[semantic sub-task]] that it cannot execute, it explicitly catches this behavior. Instead of failing, the task is then handed off to the [[LMulator]], which is the [[language model]] itself, to simulate the expected output of that specific line or function. The LM makes a linguistic prediction for the semantic sub-task (e.g., `is_sarcastic(sentence)`) and returns the result, which then updates the overall [[program state]].

This hybrid approach allows [[CoC]] to leverage the strengths of both worlds: precise [[algorithmic computations]] are offloaded to the interpreter, while complex [[semantic sub-tasks]] that are hard to formalize in code are handled by the LM's reasoning capabilities. This differs from prior work like [[Chain of Thought]] (pure natural language), [[ScratchPad]] (LM as interpreter for *all* steps), and [[Program of Thoughts]] (generates *executable* code), by explicitly integrating and switching between formal execution and linguistic simulation.

# Dataset / Experimental Setup

The primary dataset used for evaluation is [[BIG-Bench Hard]], a collection of challenging tasks designed to test advanced reasoning abilities in [[language models]]. Experiments were conducted to demonstrate the applicability of [[CoC]] across a wide variety of numerical and semantic reasoning questions. The setup involved comparing [[CoC]] against baselines such as [[Chain of Thought]] and other methods. The authors also investigated the necessity of both code interpreter execution and [[LMulator]] simulation for optimal performance, and how the approach scales with different model sizes. Further experiments explored [[CoC]]'s robustness to prompt variation, its complementarity with instruction-tuned chat models, and its potential applicability beyond language reasoning, such as in robotics.

# Key Results

- [[Chain of Code (CoC)]] achieved 84% [[accuracy]] on [[BIG-Bench Hard]] tasks, representing a significant gain of 12% over [[Chain of Thought]].
- [[CoC]] outperformed average human raters overall on [[BIG-Bench Hard]].
- On an algorithmic subset of tasks within [[BIG-Bench Hard]], [[CoC]] even surpassed the performance of the best human raters.
- The approach sets a new state-of-the-art for [[BIG-Bench Hard]] tasks.
- Both the [[code interpreter]] execution and the [[LMulator]]'s [[language model]] execution simulation were found to be necessary for achieving this high performance.
- Unlike some [[prompting]] techniques that only emerge at scale, [[CoC]] demonstrated strong performance and scalability with both large and small [[language models]].
- [[CoC]] can serve as a general-purpose reasoner, evidenced by its performance in cross-task prompting benchmarks where prompts from different problem families were used as context.
- The method is complementary to more advanced instruction-tuned chat models and robust against prompt variation.
- [[CoC]]'s applicability extends beyond language reasoning domains, with potential uses in areas like robotics.

# Limitations

The paper primarily focuses on the benefits and strong performance of [[Chain of Code]]. While no explicit limitations are detailed by the authors, a potential limitation could be the inherent reliance on the [[language model]]'s ability to accurately "emulate" or simulate the output of complex [[semantic sub-tasks]]. If the LM struggles with a particular semantic function, the entire reasoning chain could be compromised, highlighting a dependency on the LM's underlying semantic understanding capabilities.
