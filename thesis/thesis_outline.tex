%% ── Abstract ─────────────────────────────────────────────────
\begin{abstract}
Personal knowledge management (PKM) tools such as Obsidian have become central to academic knowledge work, treating vaults as personal knowledge graphs. However, the manual effort required to extract metadata and link documents creates a bottleneck. This thesis investigates whether an LLM-agent pipeline can automate the creation of structured, interlinked notes and whether this structure improves retrieval quality. We compare a standard flat RAG baseline against an agent-generated structured note system. Evaluation focuses on retrieval performance (Recall, Precision, MRR), metadata accuracy, and the quality of semantic inter-note links. Additionally, generated summaries are evaluated by a domain expert for factual correctness and utility. 

\vspace{0.5cm}
\noindent \textbf{Keywords:} RAG, Personal Knowledge Graphs, Obsidian, LLM Agents, Information Retrieval, Metadata Extraction.
\end{abstract}

\section{Introduction}
\label{sec:intro}

\subsection{Context and Motivation}
Academic research is increasingly characterized by information overload. Researchers must synthesize knowledge across hundreds of papers, leading to the adoption of tools like Obsidian that build personal knowledge graphs through atomic notes and links. However, the manual overhead of extraction and linking often leads to unstructured, isolated notes. 

\subsection{Problem Statement}
It remains an open question whether converting academic PDFs into a structured, interlinked Markdown format improves information retrieval. This thesis conducts a controlled computational study comparing a standard flat RAG baseline against a proposed LLM-agent pipeline that generates structured Obsidian notes with semantic links.

\subsection{Research Questions}
\label{sec:rqs}
\noindent \textbf{Main Question:} To what extent can an LLM agent improve retrieval in an Obsidian-style knowledge base by generating structured metadata and inter-note links, compared to a simple RAG baseline?

\begin{itemize}
    \item[\textbf{RQ1}] Does structure improve retrieval quality (Recall, Precision, F1, MRR)?
    \item[\textbf{RQ2}] How accurately does the agent extract structured metadata?
    \item[\textbf{RQ3}] How accurately does the agent generate inter-note link relation types?
    \item[\textbf{RQ4}] How does the supervisor rate summary quality (factual correctness and utility)?
\end{itemize}

\subsection{Contributions}
This thesis provides: (1) An LLM-agent pipeline for structured Obsidian note generation; (2) An empirical comparison of flat vs. structured retrieval; (3) A rigorously annotated dataset of gold labels; and (4) An open-source reproducibility framework.

\section{Related Work}
\label{sec:related}
\subsection{Retrieval-Augmented Generation}
Standard RAG relies on flat chunking and dense vector retrieval. While effective for simple lookups, it struggles with synthesis queries across documents.

\subsection{Graph-Structured RAG}
Systems like GraphRAG and HippoRAG demonstrate that structural relationships significantly improve synthesis and multi-hop reasoning.

\subsection{Personal Knowledge Graphs}
Obsidian facilitates the manual creation of PKGs. This thesis bridges the gap between automated agentic extraction and the personalized structure of an academic vault.

\section{Methodology}
\label{sec:methodology}

\subsection{Research Design}
A controlled computational experiment where the independent variable is the retrieval architecture (Flat vs. Structured) and the dependent variables are retrieval and extraction metrics.

\subsection{Dataset Construction}
\subsubsection{Paper Selection}
A purposive sampling strategy is used to construct a corpus of academic papers from arXiv, focused on the domain of AI and Knowledge Graphs. The corpus size is chosen to be sufficiently diverse for semantic linking while remaining manageable for manual "gold standard" annotation. A subset of papers is selected to match an existing expert vault by Emile van Krieken.

\subsubsection{Text Extraction}
Text is extracted using PyMuPDF, with cleaning steps for hyphenation and whitespace normalization.

\subsubsection{Query Set Design}
A dataset of 50--100 queries is constructed, stratified by difficulty: simple factual lookups, cross-paper comparisons, and synthesis queries.

\subsection{Systems Compared}
\subsubsection{Baseline RAG System}
The baseline system represents a standard, industry-practice naive RAG pipeline. It takes the extracted academic PDFs and applies a fixed-size chunking strategy (300 tokens per chunk with a 50-token overlap). These chunks are embedded using the \texttt{all-MiniLM-L6-v2} sentence-transformer model and stored in a dense cosine similarity index.

\subsubsection{Proposed: LLM-Agent Structured Note System}
A multi-agent pipeline (Metadata, Note, and Link Agents) transforms PDFs into Obsidian Markdown. Retrieval is performed on document-level vectors of the full structured notes.

\subsection{LLM Configuration}
All agents use the \textbf{Gemini 1.5 Flash} model (via the stable \texttt{gemini-flash-latest} API model channel). Using this model provides a generous daily free-tier request quota (1,500 requests per day) compared to preview model channels (e.g., 2.0 and 2.5), enabling complete end-to-end execution of the pipeline. For extraction tasks (Metadata and Link Agents), the temperature is set to \textbf{0.0} to ensure deterministic, low-variance output. For the Note Agent (summary generation), a temperature of \textbf{0.3} is used to allow for more natural synthesis while maintaining factual grounding.

\subsection{Evaluation Data}
The system is evaluated against a "Gold Standard" created manually:
\begin{itemize}
    \item \textbf{Relevance Labels:} A binary mapping of which papers are actually relevant to each query in the query set.
    \item \textbf{Metadata Gold Labels:} Manually verified fields (authors, venue, results) for every paper in the corpus.
    \item \textbf{Link Gold Labels:} A mapping of the "true" semantic relationships (e.g., \textit{extends}, \textit{usesMethod}) between papers.
    \item \textbf{Summary Sample:} A random sample of 20 generated summaries is selected for expert human evaluation by the supervisor.
\end{itemize}

\section{Evaluation Metrics}
\label{sec:metrics}

\subsection{Retrieval Quality Metrics}
We choose the following metrics because they reflect the primary user goals in a Personal Knowledge Base:

\begin{itemize}
    \item \textbf{Recall@k:} Measures the system's ability to find all relevant documents.
    \begin{equation}
        \text{Recall@}k = \frac{|\text{Retrieved}^{(k)} \cap \text{Relevant}|}{|\text{Relevant}|}
    \end{equation}
    
    \item \textbf{Precision@k:} Measures the fraction of retrieved documents that are relevant.
    \begin{equation}
        \text{Precision@}k = \frac{|\text{Retrieved}^{(k)} \cap \text{Relevant}|}{k}
    \end{equation}

    \item \textbf{F1@k:} The harmonic mean of Precision and Recall.
    \begin{equation}
        \text{F1@}k = 2 \cdot \frac{\text{Precision@}k \cdot \text{Recall@}k}{\text{Precision@}k + \text{Recall@}k}
    \end{equation}

    \item \textbf{MRR (Mean Reciprocal Rank):} Measures ranking quality.
    \begin{equation}
        \text{MRR} = \frac{1}{|Q|} \sum_{i=1}^{|Q|} \frac{1}{\text{rank}_i}
    \end{equation}
    where $\text{rank}_i$ is the position of the first relevant document for query $i$.
\end{itemize}

\subsection{Extraction & Link Metrics}
Metadata is evaluated via \textbf{Exact Match} for discrete fields and \textbf{F1-score} for multi-label fields (e.g., topics). Link quality is evaluated using Precision and Recall of the generated link set against the gold standard.

\subsection{Alternative Metrics for Discussion}
In the development of this evaluation framework, several alternative metrics were considered. Understanding why these were not selected provides important context for the rigour of this study:

\begin{itemize}
    \item \textbf{NDCG (Normalized Discounted Cumulative Gain):} 
    \textit{What it is:} A measure of ranking quality that accounts for "graded relevance" (e.g., a document can be "highly relevant" or "partially relevant").
    \textit{Why not selected:} NDCG requires multi-level relevance labels. For a Bachelor thesis, binary relevance (relevant vs. non-relevant) is more manageable for manual annotation and ensures higher inter-annotator agreement if spot-checked.
    
    \item \textbf{ROUGE and METEOR:}
    \textit{What they are:} Automated metrics that measure the n-gram overlap between a generated summary and a reference summary.
    \textit{Why not selected:} These metrics correlate poorly with the semantic accuracy of academic text. A summary could have high ROUGE overlap but hallucinate a key numerical result, making it useless. We prioritized **Expert Human Evaluation** to ensure scientific validity.
    
    \item \textbf{LLM-as-a-Judge:}
    \textit{What it is:} Using a powerful model (like GPT-4o) to rate the quality of outputs from a smaller model.
    \textit{Why not selected:} LLMs often exhibit "self-preference bias" (favouring styles similar to their own) and can be inconsistent. Relying on **Gold Standard** human labels remains the "ground truth" in IR research.
    
    \item \textbf{Hit Rate (Success at k):}
    \textit{What it is:} A binary measure of whether *at least one* relevant document appears in the top-k.
    \textit{Why not selected:} While useful for simple Q\&A, it is too lenient for research work where the user often needs to find *all* related papers to conduct a literature review. Recall@k is a more rigorous substitute.
\end{itemize}
