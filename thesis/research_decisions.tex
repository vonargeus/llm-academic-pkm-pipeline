\section*{Frequently Asked Questions \& Conceptual Clarifications}
\label{sec:research_decisions}

This document compiles the conceptual questions, research design justifications, and methodology clarifications raised during the development of the Bachelor of AI thesis evaluation pipeline.

\begin{enumerate}[label=\textbf{Q\arabic*:}, leftmargin=*]
    \item \textbf{Why does the agentic pipeline output Markdown notes rather than native PDFs?} \\
    LLMs generate text sequentially and cannot natively output formatted binary PDF files. Markdown is a lightweight, plain-text format that LLMs write with high fidelity. Furthermore, Markdown supports structured headers and YAML frontmatter (enabling structured key-value metadata), as well as native Obsidian-style wiki-links (\texttt{[[Target Note]]}) to model graph edges. Raw PDF text lacks these semantic wrappers, making it unstructured layout data.

    \item \textbf{Why is the embedding model \texttt{all-MiniLM-L6-v2} used for all configurations, and how do their embedding processes differ?} \\
    Using the same embedding model across all systems is a critical scientific control. If we used a different model for Structured RAG, the performance delta could be caused by the model rather than the document structure. 
    In terms of execution:
    \begin{itemize}
        \item \textit{Flat RAG} embeds raw 300-character sequential chunks.
        \item \textit{Structured RAG} embeds the full note (YAML frontmatter + summary text) as a single vector, capturing global context.
        \item \textit{Link-Expanded RAG} uses the same note embeddings to generate seeds, then adds the topological boost ($\alpha$) during the re-ranking step.
    \end{itemize}

    \item \textbf{Why are the Structured and Link-Expanded RAG configurations evaluated separately instead of permanently merged?} \\
    In production systems, they are merged. However, in an academic thesis, they must be separated to conduct a proper ablation study. By evaluating them separately, we can isolate the individual performance gains of summarization/structure (Structured RAG) versus topological link propagation (Link-Expanded RAG), revealing the specific trade-offs at different values of $k$.

    \item \textbf{What are the formal criteria for defining data as ``structured'' versus ``unstructured'' in this research?} \\
    Unstructured data has no predefined schema or semantic labeling (e.g., raw PDF text streams). Structured data adheres to a formal data model or schema. In this project, the Markdown notes are defined as structured because they contain a queryable YAML frontmatter schema, standardized section headings, and explicit typed link predicates (\texttt{[[WikiLinks]]}).

    \item \textbf{Why do we retrieve a larger number of chunks ($K_{chunk} = 150$) for Flat RAG to get 10 unique papers, and why must we evaluate exactly 10 unique papers?} \\
    In Information Retrieval (IR) science, systems are evaluated on a ranked list of recommendations (e.g., top-10) using metrics like Recall@10 and Mean Reciprocal Rank (MRR). 
    Because Flat RAG chunks are tiny (300 characters, ~50 words) to prevent model truncation, a single paper is split into over 100 chunks. Highly relevant papers cluster at the top of the vector search, meaning the top 10 retrieved chunks might all belong to the same 1 or 2 papers. To construct a list of 10 unique candidate papers for a fair document-level comparison, Flat RAG must retrieve a larger chunk pool (empirically calibrated at an average of 146.88 chunks) before deduplicating by parent document ID.

    \item \textbf{How does the embedding model interact with the LLM in Structured RAG?} \\
    There is a strict separation of roles:
    \begin{itemize}
        \item \textbf{Generation Phase (LLM):} The LLM (Gemini API) is used once as an offline preprocessing step to read raw PDF text and write structured Markdown notes (summaries, metadata, wiki-links).
        \item \textbf{Retrieval Phase (Embedding Model):} During the actual search experiment, the LLM is not used. We embed the generated Markdown notes using the local `all-MiniLM-L6-v2` embedding model. Cosine similarity is calculated directly between the query vector and note vectors to find the top matches.
    \end{itemize}

    \item \textbf{What is confounding bias, and how does it manifest in this evaluation?} \\
    Confounding bias occurs when a third, unmeasured variable correlates with both the proposed cause and the proposed effect, leading to a false or exaggerated causal attribution. 
    In our experiment, the confounding variable is \textbf{document length/noise}. Shorter, condensed summaries (structured notes) naturally present less text noise to dense vector retrievers than long raw PDF chunks. This difference in length is a confounder because retrieval improvements could be partially driven by the brevity of the summaries rather than their metadata structure. We document this in the ``Limitations'' section to maintain scientific transparency.
\end{enumerate}
